# 2026-03-13: make Symphony auto loop strictly sequential
- **Unexpected hurdle:** The TUI would relaunch as soon as the periodic refresh loop reported a `ready` state even when the previous run was still finishing, so the auto loop sometimes started a second Ralph job before the server cleared the first `activeRun` slot.
- **Diagnosis path:** Traced `runAutoLoopCycle` and `waitForActiveRunCompletion` in `scripts/symphony-tui.js` and saw that the cycle only inspected the shared `lastSymphonyStatus` cache, so timing could let a stale `ready` state fall through before an explicit refresh confirmed the run was gone.
- **Chosen fix:** `runAutoLoopCycle` now keeps the UI explicit about searching versus waiting, refreshes immediately after launch to capture the new run ID, then holds in `waitForActiveRunCompletion` while polling `refreshLoop` until the recorded ID disappears; the helper only schedules the next cycle after seeing that completion, so launches remain sequential.
- **Next-time guidance:** When guarding sequencing, tie the guard to a refreshed payload (not just cached status) and show the waiting/searching distinction in the UI so it is obvious when the loop is paused for completion.
- **Validation:** `npm test` passed. Live verification against a fresh local Symphony server also passed: after pressing `A` in `node scripts/symphony-tui.js`, the TUI launched `2026-03-13T21:33:16.750Z--dadeto-7gxw`, switched to `Auto: waiting`, and six seconds later the TUI plus `curl http://127.0.0.1:4322/api/symphony/status` still showed the same `activeRun.runId` with no second launch attempt, proving the auto loop stayed strictly sequential while the first Ralph run remained active.
