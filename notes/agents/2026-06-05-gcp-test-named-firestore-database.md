The live Admin SDK path was therefore landing on `(default)` instead of the `t-...` database, which matched the runtime error.
- Chosen fix: changed `src/core/cloud/firestore-helpers.js` to call `getFirestoreFn(undefined, databaseId)` for named databases without an explicit app, and tightened the Firestore contract tests plus the generate-stats wrapper test to expect the two-argument form.
- Next-time guidance: when a Cloud Run log mentions a database mismatch, check the exact Admin SDK call shape first. A one-argument `getFirestore` mock can hide the difference between “named database” and “default database” in unit tests.
