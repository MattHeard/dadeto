# Dadeto Overview

My view of Dadeto is that it should be a modular integration substrate for uncertain futures.

## Core thesis

I think the durable thing is not one software production mode replacing the others, but a system that can host and move between several modes over time:

1. fixed software built once for many users
2. configurable or modular software built from reusable parts
3. custom software generated or assembled on demand
4. model-mediated software where a generative model participates in the runtime or production loop

Dadeto should make those modes interoperable instead of treating them as separate worlds.

## What matters

- Modularity matters because it preserves optionality.
- Decomposition matters when it lowers future replacement and recomposition cost.
- Some modules may eventually be internalized by models, but the architecture should support migration rather than assume a final form.
- The system should be discoverable and configurable by AI coding agents first, while still remaining legible enough for human oversight.

## How I want it to sell

Dadeto should aim for present-tense, low-friction wins rather than speculative future bets.

- The customer-facing wedge should feel obviously better on the dimensions the customer actually cares about.
- GTM should not depend on shared beliefs about the future or long-dated speculative trade-offs.
- I want reversible, measurable, win-win changes rather than extractive partnerships or “bet on the future” deals.
- Legacy organizations are a strong target because they already contain useful capability trapped in rigid systems.

## How I want to build it

My building loop is recursive:

1. solve a real problem
2. notice repeated structure or duplication
3. extract reusable modules
4. expose new seams for recomposition
5. make the next use case cheaper because the substrate is better

The goal is not just to ship features. The goal is to turn feature work into substrate-building work.

## Pricing

I want the pricing model to preserve economic optionality the same way the architecture preserves technical optionality.

- Keep fixed costs low.
- Anchor pricing to variable cloud or supplier costs where possible.
- Offer stable or locked pricing by layering hedges on top of the variable base.
- Refuse to bet long-term on any single unit price.

The customer-facing packaging should stay simple even if the underlying cost model is more sophisticated.

## Hardware

I also want Dadeto to extend beyond software into physical systems.

- Start with the least invasive viable integration.
- Prefer observation before control.
- Use smart devices, screens, sensors, and other low-friction inputs to build a mental model of legacy hardware integration.
- The world is not software-only, so legacy physical systems should be treated as possible integration targets, not exceptions.

## Short version

Dadeto should help me build systems that are modular, agent-legible, adaptable, and cheap to change, while still delivering obvious present-day value.
